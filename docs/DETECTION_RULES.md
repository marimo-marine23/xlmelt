# xlmelt 構造検出ルールブック

xlmeltはExcelファイルのセルメタデータ（フォント、罫線、結合、背景色、number_format）を使い、LLMなしで文書構造を検出します。このドキュメントでは、ルールベース検出の**できること・できないこと・LLMで改善できること**を整理します。

---

## 検出の流れ

```
Excel → セルメタデータ読み取り → 行単位の分類 → 後処理（罫線なしテーブル・フッター統合）→ 連続行のグループ化 → セクション生成
```

各行は以下の優先順で分類されます（先にマッチしたものが採用）:

1. **空行** — 値のあるセルが0個
2. **見出し** — 単一値セル + 大フォント/太字結合/太字中央揃え/太字短文
3. **テーブル行** — 罫線付きセルが50%超 + 値セル2個以上
4. **キーバリュー行** — 2〜6セル + 太字/非太字の混在 or 背景色の混在
5. **リスト項目** — 1〜2セル + リストマーカー（`・` `-` `●` `1)` `(1)` 等）で始まる ✨Phase 2a
6. **テキスト** — 上記いずれにも該当しない行

### 後処理パイプライン ✨Phase 2a

分類後、以下の後処理が順に適用されます:

1. **空行吸収**: テーブル行に挟まれた1行の空行 → `empty_between_table`
2. **罫線なしテーブル検出**: 3行以上の連続テキスト行で列配置とデータ型に一貫性 → テーブル行に昇格
3. **テーブルフッター統合**: テーブル直後の「合計」「小計」等を含む行 → テーブル行に統合
4. **暗黙的行インデックス補完**: テーブルデータの先頭1〜2列で空白セル → 上の行の値で補完

---

## 1. 見出し検出

### ルール

| 条件 | 検出レベル | 例 |
|---|---|---|
| フォントサイズ ≥ 18pt | H1 | タイトル行 |
| フォントサイズ ≥ 14pt | H2 | 章見出し |
| 太字 + フォント ≥ 12pt | H3 | 節見出し |
| 太字 + 結合幅 ≥ 列数の1/3 | H2 | 幅広結合見出し |
| 太字 + 中央揃え | H3 | 中央揃え見出し |
| 太字のみ + テキスト40文字以内 | H4 | セクション見出し（「1. 移転先情報」等） |

### 前提条件

- **その行に値のあるセルが1つだけ**であること（結合セルは1つとカウント）
- 複数セルに値がある行は見出しとして検出されない

### できないこと

| ケース | 理由 |
|---|---|
| フォントサイズもスタイルも同じで、テキスト内容のみ「第1章」等で区別している見出し | テキスト内容は解析しない |
| 番号付きリストの先頭行（太字なし） | スタイル的に区別できない |
| 複数セルにまたがる見出し（結合されていない） | 「単一セル」条件を満たさない |

---

## 2. テーブル検出

### ルール

- 行内の**占有列範囲**（値またはborderを持つセルのmin〜max列）内の**結合元セル**のうち**50%超**が罫線（上下左右いずれか）を持つ → テーブル行
- 占有列範囲外の空セルは計算に含めない（方眼紙等で先頭/末尾の空列による希釈を防止）
- テーブル行が連続している区間を1つのテーブルとしてグループ化
- テーブル行の間に挟まれた1行の空行は吸収される（`empty_between_table`）

### 罫線なしテーブル検出 ✨Phase 2a

罫線がなくても、以下の条件を満たすテキスト行の連続は**テーブルとして検出**:

| 条件 | 閾値 |
|---|---|
| 連続行数 | ≥ 3行 |
| 各行のセル数 | ≥ 2セル |
| セル数の一貫性 | 最頻セル数が全行の60%以上 |
| 列位置の一貫性 | 60%以上の行で使われる列が2列以上 |
| データ型一貫性（強シグナル） | いずれかの列で50%以上が数値 |
| 列位置一貫性（弱シグナル） | 80%以上の列が一貫 + セル数≥3 |

### テーブルフッター統合 ✨Phase 2a

テーブル直後の行が以下の条件を満たす場合、テーブルの最終行として統合:

- **フッターキーワード**: 合計、計、小計、総計、Total、平均、Average 等
- **罫線一致**: テーブルと同じ罫線スタイルを持つ行

### 暗黙的行インデックス補完 ✨Phase 2a

テーブルデータの先頭1〜2列で:
- セルが空文字列（結合プレースホルダーではない）
- 同列の上の行に非空値がある
- 他の列にデータがある（完全空行ではない）

→ 上の行の値をコピーして補完。日本のExcelでよくある「グループ化」パターンに対応。

### 横方向テーブル分離

- テーブル行グループ内で**値が存在する列**を検出
- 連続する列をグループ化し、**空白列で区切られた列グループ**を別テーブルとして分離
- 同じ行範囲に複数テーブルがある場合、`layout_group` IDを付与して横並び関係を記録
- HTMLでは `<div class="table-group">` + flexboxで横並びレンダリング

### ヘッダー検出（4段階ヒューリスティック）

テーブル先頭から連続する行を以下の4つの手がかりで判定:

| 優先度 | ヒューリスティック | 内容 |
|---|---|---|
| 1 | **スタイル判定** | 全ての結合元セルが太字 or 背景色あり |
| 2 | **colspan結合** | 行内にcolspan結合がある（マルチレベルヘッダーの指標） |
| 3 | **rowspanパターン** | 次の行にrowspan結合が始まっている（データ行の指標→ヘッダー境界） |
| 4 | **データ型一貫性** | 対象行がテキスト中心、次の行が数値中心（ヘッダー→データの切り替え） |

### マルチインデックス（結合セル）

- ヘッダー内の結合セル → `colspan` として記録
- データ行内の結合セル → `rowspan` として記録
- 結合されたセルの被結合部分 → `null` として出力

### できないこと

| ケース | 理由 | 影響度 |
|---|---|---|
| ピボットテーブルの構造 | 特別な検出ロジックなし | 低 — 通常の表として検出される |
| 結合なしで3列以上が暗黙階層の行 | 先頭2列のみ補完 | 低 — 3列以上の階層は稀 |

## 3. リスト検出 ✨Phase 2a

### ルール

行内のセルが1〜2個で、最初のセルの値が以下の正規表現にマッチ → リスト項目:

```
^\s*(
  [・●■◆◇▶▷▸▹★☆○►]         # 記号マーカー
  |[-–—•‣◦]                    # ダッシュ・Unicode箇条書き
  |\d{1,3}[.)）]                # 番号: 1. 1) 1）
  |[①②③④⑤⑥⑦⑧⑨⑩...]          # 丸数字
  |[ⅰⅱⅲⅳⅴ...ⅠⅡⅢ...]          # ローマ数字
  |[a-zA-Z][.)）]               # 英字: a. a)
  |[(（]\d{1,3}[)）]            # 括弧付き番号: (1) （1）
  |[(（][a-zA-Z][)）]           # 括弧付き英字: (a) （a）
  |※                            # 注記マーカー
)\s*
```

連続するリスト項目は1つの `LIST` セクションにグループ化。マーカーはテキストからstripされ、出力は `{"ordered": bool, "items": [str]}` 形式。番号型マーカー（`1.` `(1)` `①` 等）は `ordered: true`、記号型マーカー（`・` `●` `-` 等）は `ordered: false` として分類。HTMLでは `<ol>` / `<ul>` を使い分け。

### できないこと

| ケース | 理由 |
|---|---|
| マーカーなしのリスト（インデントのみ） | テキスト内容のパターンに依存 |
| ネストされたリストの階層 | インデントレベルの解析は未実装 |
| 3セル以上にまたがるリスト項目 | テーブルとの誤判定を避けるため除外 |

---

## 4. キーバリュー検出

### ルール

- 行内に2〜6個の値セル
- **条件A**: 太字セルと非太字セルが混在 → 太字=キー、非太字=値
- **条件B**: 背景色ありセルと背景色なしセルが混在（4セル以下） → 色付き=キー

### できないこと

| ケース | 理由 |
|---|---|
| 全セルが同じスタイル（太字なし・背景色なし）のラベル+値ペア | スタイル差がないと判別不能 |
| 7セル以上の行 | テーブルとの誤判定を避けるため上限あり |

---

## 5. テキスト検出

### ルール

- 上記のいずれにも該当しない行 → テキスト
- 連続するテキスト行は1つのテキストセクションに結合
- 行内の複数セルの値は2スペースで結合

---

## 6. Excel方眼紙検出 ✨Phase 2a 強化

### ルール

- 列数 ≥ 10 かつ 列幅の平均 ≤ 5 かつ 列幅の分散 < 0.5 → 方眼紙パターン

### 方眼紙前処理 ✨Phase 2a

方眼紙と判定されたシートでは、通常の構造解析の前に**論理セル結合**を実施:

1. 各行で値のあるセルを検出
2. 値セルの後に続く空セル列を、値セルの `merge_width` として拡張
3. 次の値セルの直前まで結合幅を拡張

これにより、方眼紙上で結合を使わずに配置されたコンテンツが、通常のセルと同様に処理可能。

### できないこと

- 方眼紙上でのフリーフォームレイアウトの意味的解釈
- 結合セルの配置パターンから「フォーム」や「帳票」を推定すること

---

## 7. 数値フォーマット型推定 ✨Phase 2a

### ルール

Excelの `number_format` 文字列からセマンティックな型を推定し、JSON出力に `"format"` フィールドとして付与:

| number_format パターン | 推定型 | 例 |
|---|---|---|
| `yy`, `mm/dd`, `yyyy` 等（時刻なし） | `date` | `yyyy/mm/dd` |
| `yy` + `h:mm` 等 | `datetime` | `yyyy/mm/dd hh:mm` |
| `h:mm`, `hh:`, `:ss` 等 | `time` | `hh:mm:ss` |
| `%` を含む | `percentage` | `0.0%` |
| `¥`, `$`, `€`, `£`, `円` 等 | `currency` | `¥#,##0` |
| 小数点 + 数字/# | `decimal` | `0.00` |
| `#,##` or `0,0` | `number` | `#,##0` |
| `General` or なし | なし（省略） | — |

---

## 8. 画像・グラフ ✨Phase 2a 強化

### ルール

- .xlsx内のZIP構造から`xl/media/`以下の画像ファイルを抽出
- **シートマッピング**: `xl/worksheets/_rels/` のリレーションXMLを解析して、画像をシートに紐付け ✨Phase 2a
- **アンカー位置**: `xl/drawings/` のDrawing XMLを解析して、画像のアンカーセル位置を特定 ✨Phase 2a
- **命名規則**: `{シート名}_img{連番}.{拡張子}` 形式で保存 ✨Phase 2a

### できないこと

- 画像の内容理解（OCR, 図表認識）
- グラフオブジェクトのデータ抽出（チャートの元データ取得）

---

## 9. .xls (旧形式) 対応 ✨Phase 2a

### ルール

- ファイル拡張子が `.xls` の場合、xlrdライブラリを使用してパース
- xlrdの `formatting_info=True` でフォント・罫線・背景色・number_formatを取得
- openpyxlベースのパーサーと同じ `CellInfo` データモデルに変換
- 以降の構造解析は .xlsx と完全に同じロジックを使用

### できないこと

- .xls 形式の画像抽出（OLE compound document のパースが必要）
- マクロ付き .xls のマクロ内容解析

---

## LLMで改善できること

ルールベースの限界を補完するために、LLMを使うことで以下の改善が期待できます。

### 高い効果が見込めるもの

| 改善項目 | 内容 | なぜルールベースでは無理か |
|---|---|---|
| **罫線なしテーブルの完全検出** | Phase 2aの部分対応では拾えない曖昧なパターン | 文脈とレイアウト意図の理解が必要 |
| **誤分類の修正** | KVとテーブルの判別ミス、見出しの見逃しを修正 | 閾値ベースの判定にはグレーゾーンがある |
| **セクションの階層構造** | 「第1章 > 1.1 > 1.1.1」の親子関係を推定 | テキスト内容のパース + 文脈理解が必要 |
| **テーブル隣接コンテンツの意味的統合** | 小計行・注釈行の意味的な関連付け | フッターキーワード以外の関連性判断が必要 |

### 中程度の効果

| 改善項目 | 内容 |
|---|---|
| 方眼紙レイアウトの解釈 | 結合パターンから帳票フォームの構造を推定 |
| ネストされたリストの階層化 | インデント+テキスト内容からリスト階層を推定 |
| 文脈的見出し検出 | テキスト内容に基づく見出し判定（「第1章」等） |

## まとめ：ルールベース vs LLM

```
         ルールベース（無料）              LLM補正（従量課金）
         ─────────────              ─────────────
検出対象  罫線テーブル ✅              罫線なしテーブル（完全） ✅
         罫線なしテーブル（部分） ✅    文脈的見出し ✅
         太字見出し ✅               同スタイルKV ✅
         スタイル差KV ✅              暗黙階層（3列以上） ✅
         結合セル ✅                 画像内容理解 ✅
         箇条書きリスト ✅            ネストリスト ✅
         暗黙行インデックス ✅
         テーブルフッター ✅
         画像抽出+シート紐付 ✅
         number_format型推定 ✅
         .xls旧形式 ✅
         方眼紙前処理 ✅

精度     75-90%（構造化されたExcel）    90-95%（推定）
         55-70%（自由レイアウト）       85-90%（推定）

コスト    0円                       ファイルあたり数円〜数十円
```

ルールベースだけで十分な場合:
- **罫線付きの表が中心**の仕様書、台帳、管理表
- **太字見出し**で章立てされた文書
- **定型フォーマット**の帳票
- **箇条書きリスト**を含む報告書

LLM補正が有効な場合:
- **罫線なし・自由レイアウト**のExcel
- **構造が暗黙的**で見た目の手がかりが少ない文書
- **高精度が求められる**バッチ変換
